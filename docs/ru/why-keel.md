# Почему Keel, и что изменилось после SkillForge

Keel — преемник SkillForge: та же задача, противоположный вывод. SkillForge пытался дать агенту целую систему: свой MCP-сервер,
векторный индекс по памяти проекта, реранкер, агентов-персон, гейты
согласования, артефакты на каждую задачу, апдейтер. Keel оставляет четыре
вещи, которые действительно несли нагрузку, и выбрасывает остальное.

SkillForge я закопал; этот документ — обоснование: что было измерено, что
показал прод и, не менее важно, что не измерялось вовсе. Каждое число ниже
воспроизводится из двух деревьев; команды — в конце.

## Замеры

Сравниваются SkillForge 1.8.2 в том виде, в каком он был развёрнут в
проекте ShipPulse, и Keel 1.6.0. Токенные цифры — оценка из расчёта ~4
символа на токен и помечены как оценка; всё остальное — точный подсчёт.

| | SkillForge 1.8.2 | Keel 1.6.0 | Разница |
|---|---|---|---|
| Всегда загруженный контракт | `CLAUDE.md` (80 строк) + `_protocol.md` (247 строк) = 327 строк / 15 245 символов (~3,8k токенов, оценка) | `CLAUDE.md` = 81 строка / 4 227 символов (~1,1k токенов, оценка) | в 3,6 раза меньше |
| Код ядра, которому надо доверять и который надо поддерживать | 6 316 строк TS/JS/shell в 17 файлах — MCP-сервер, конвейер эмбеддингов, eval поиска, sync, update, verify | 1 701 строка shell в 10 файлах — установщик, сборщик архива, 3 хука, safe-run, зачистка миграции, граф памяти, якоря по коду, самотесты ядра | в 3,7 раза меньше |
| Требуемые сервисы в рантайме | 2 — MCP-сервер (bun) + демон Ollama с эмбеддингами `bge-m3` | 0 | — |
| Занимаемое место после установки | 63 МБ / 3 796 файлов (24 МБ — `node_modules` MCP-сервера; остальное — векторный индекс и `.tgz`-архивы, которые апдейтер делал сам себе). Чистое поставляемое дерево без этого: 4,3 МБ / 133 файла | 312 КБ / 54 файла | в 14 раз меньше самой щадящей меры; в 207 раз меньше того, во что он превращается в рабочем проекте |
| Сабагенты | 8 персон — orchestrator, product, qa, security, devops, research, copy, skill-creator | 2, по изоляции контекста — `scout` (разведка только на чтение), `verifier` (независимый судья) | — |
| Поиск по памяти | эмбеддинги (`bge-m3` через Ollama) + реранкер + PageRank по графу заметок | индекс `MEMORY.md` + `grep`, плюс якоря по коду (`recall`) | — |
| Золотой набор, ради которого всё это строилось | 6 запросов, насыщение на recall@5 = 1.0 | — | — |
| Церемония до первой строки кода | протокол + `plan.json` + цитаты `because[]` + гейт согласования — одинаково для переписывания архитектуры и для правки отступа | 0 файлов на малой задаче; 2 файла (бриф, отчёт) на большой | — |

Ядро растёт от версии к версии, и таблица этого не скрывает. В 1.3.0 было
893 строки shell в 5 файлах — перевес в 7,1 раза. В 1.4.0 добавились
зачистка миграции, проверка обновлений и граф памяти: 1 305 строк в 8
файлах, перевес упал до 4,8 раза. В 1.5.0 пришли якоря по коду: 1 460 строк
в 9 файлах, перевес 4,3 раза. В 1.6.0 — самотесты ядра: 1 701 строка в 10
файлах, перевес 3,7 раза. Из 241 строки последнего прироста 157 — сам
тестовый набор: код, который увеличивает счёт «кода, которому надо
доверять», и при этом делает каждую остальную строку надёжнее. Это
противоречие здесь названо, а не спрятано; к тому же 6 316 строк SkillForge
включали его собственный eval-харнесс, так что тесты Keel посчитаны по тем
же правилам. Каждая добавленная строка — строка, которой кому-то придётся
доверять, а сравнение, которое только улучшается, — сравнение, которым
управляют.

Ядро теперь тестирует само себя. `test/run.sh` — bash-ассерты поверх
поставляемых скриптов (`anchors.sh`, `graph.sh`, `sweep.sh`,
`update-check.sh`): одноразовые фикстуры, офлайн, ненулевой код выхода при
любом падении. `build-archive.sh` прогоняет их первым делом и отказывается
собирать архив, если упал хоть один: скрипт, проваливший собственный тест,
в релиз не попадает. Набор появился не для галочки — каждый из трёх
последних релизов вёз реальный баг в скриптах ядра, и ловил его только
внимательный ручной разбор. `--check` не замечал переименования
`handleRequest` → `handleRequestV2`, потому что `grep -F` ищет подстроку;
блок MENTIONED ранжировался по числу совпавших строк, а не упоминаний.
Ровно эти случаи зашиты в набор, чтобы они не вернулись. Учёт честный: из
1 701 строки ядра 223 — только для мейнтейнера (`build-archive.sh` — 66,
`test/run.sh` — 157), в проект уходит 1 478 строк в 8 файлах.

## Что показал прод

Числа объясняют цену. Ущерб описан в четырёх заметках из собственной памяти
ShipPulse, написанных тогда, когда SkillForge был живой системой. Это
первоисточники, а не реконструкция задним числом.

### 1. Инфраструктура поиска убила работу, которой служила

`antipatterns/skillforge-mcp-misreports-ollama-when-direct-call-works.md`:
на macOS Apple Silicon `OLLAMA_NUM_PARALLEL=4` × Metal-хипы `bge-m3` при
пачечной эмбеддинг-нагрузке разносили VM commit, ОС убивала Ollama по OOM,
и отказ каскадом уходил в IDE-хост. 31.05.2026 это убило стейдж 009
ShipPulse на середине фан-аута и потеряло три артефакта сабагентов. Система
памяти уничтожила работу, ради запоминания которой существовала.

### 2. Рабочий паттерн системы обходил её собственный поиск

`patterns/stage-012-013-multiwave-ship-pattern.md` — паттерн самого
успешного релизного прогона SkillForge — предписывает дословно: «сабагенты
НЕ вызывают MCP-память (путь через ollama нестабилен); они грепают
`memory/**`», а заметки памяти «пишутся напрямую файлами (MCP
`memory_write` отдаёт 500 при деградировавшей ollama)». Векторный индекс,
реранкер и PageRank обходились лучшей практикой самой системы — в пользу
grep и обычных файлов. Keel — этот обход, возведённый в архитектуру.

### 3. Гейты обходили под давлением

`antipatterns/engine-bypass.md` фиксирует прогоны, где пропускали гейт
согласования и помечали юниты зелёными без артефактов, потому что «под
давлением сроков гейты ощущаются бюрократией». Давление создавала сама
конструкция, а не чья-то лень. Гейт, который обходят на важных прогонах, —
не механизм безопасности, а налог на неважные.

### 4. Персоны всасывали работу, которую должны были маршрутизировать

`antipatterns/fat-agent-prompt.md`: файлы агентов приходилось держать в
узде жёстким лимитом в 55 строк с проверкой в `verify`, потому что они
непрерывно обрастали доменной логикой и расходились с каталогом скиллов.
Роли требуют надзора, изоляция контекста — нет. Keel делит сабагентов по
контексту, а не по должности, и это совпадает с наблюдением самой
Anthropic: агенты-роли тратят на координацию больше токенов, чем на работу.

## Знание, привязанное к коду

Новое в 1.5.0 — скилл `recall`, якоря по коду.

До него заземление в памяти работало по симптому — грепнуть `memory/` по
словам из ошибки и надеяться — и никогда по месту. В реальной памяти
проекта из 222 заметок 141 называет файлы кода: 494 упоминания, 237
уникальных файлов, и ничто из этого не было достижимо со стороны кода.
Спросить, что проект знает про `apps/web/proxy.ts`, до открытия файла было
нельзя. (Команда для воспроизведения — в конце.)

Якорь — это объявление во front-matter: заметка сама говорит, о каком коде
она.

```yaml
code:
  - apps/web/proxy.ts#handleRequest
  - apps/web/middleware.ts
```

Отсюда два запроса — без скорера, без эмбеддингов, без демона; якорь либо
разрешается, либо нет:

```sh
bash .claude/skills/recall/anchors.sh apps/web/proxy.ts   # что мы знаем про этот код
bash .claude/skills/recall/anchors.sh --check             # мёртвые якоря
```

Ответ приходит двумя наборами. ANCHORED — заметки, объявившие этот код во
front-matter: точно и проверяемо. MENTIONED — упоминания в прозе,
ранжированные по плотности: полезно, но шумно, и проверить их нечем.

Детект гнили — ради него якоря и сделаны. Переименуйте символ или удалите
файл, и `--check` доложит `DEAD_SYMBOL` / `DEAD_FILE` — оба пути проверены
end-to-end: и переименование, и удаление были пойманы. Упоминание в прозе
не проверяется в принципе; в этом и разница между упоминанием и якорем.

Рёбра код→код здесь не строятся, и это осознанно. Вызывающих, ссылки и
иерархию вызовов считает serena (LSP, уже прописана в `.mcp.json` при
установке) — `find_symbol`, `find_referencing_symbols` — точно и вживую.
Структура кода, которую поддерживают руками, гниёт; LSP — нет. Слой якорей
несёт только то, чего LSP знать не может: что мы узнали про это место в
коде.

Правило 2 контракта туда и маршрутизирует. Оно отправляет модель в индекс
`memory/MEMORY.md`, велит «идти по ссылкам, которые подходят к задаче», а
теперь ещё и: «Прежде чем менять файл, который не ты только что написал,
заземлись по месту: скилл `recall`». Из-за этой строки контракт вырос с 79
строк до 81. Скилл, в который никто не маршрутизирует, — скилл, которым
никто не пользуется; ровно так у предшественника умер каталог скиллов.

### Граф заметок

Два графа легко перепутать, и в 1.4.0 этот документ их перепутал,
перепродав тот, который значит меньше. Граф заметка↔заметка — это гигиена
памяти; граф код↔знание, описанный выше, — то, что действительно помогает
с кодом.

Строка про поиск в таблице — про скорер, а не про граф. Рёбра на месте, где
были всегда: `[[wikilinks]]` внутри `memory/*.md`. В реальной памяти из 222
заметок (плюс индекс `MEMORY.md`) — 919 рёбер, 0 битых ссылок, 4 сироты;
`graph.sh` напечатает ровно эти числа, и снос MCP-сервера в них ничего не
изменил:

```sh
bash .claude/skills/memory-consolidation/graph.sh           # хабы, битые ссылки, сироты, итоги
bash .claude/skills/memory-consolidation/graph.sh --edges   # сырой список смежности
```

Хабы, битые ссылки, сироты — отчёт о состоянии заметок, честно полезный для
этого и ни для чего больше. Писать код он не помогает и не обещает.
(`memory/` — обычный markdown с `[[wikilinks]]`, так что он заодно
открывается в Obsidian или VS Code Foam. Это свойство формата, а не
возможность ядра.)

Граф SkillForge был таким же — заметка↔заметка; про код он не знал ничего.
И источником правды он не был: SkillForge пересобирал его на каждом поиске
из тех же `[[wikilinks]]` в тех же файлах (`retrieval-eval.ts`,
`buildLinkGraph`) и прогонял по нему персонализированный PageRank
(«HippoRAG-lite») как множитель-буст: `1 + 0.2 * graph` поверх
`0.8 * vector + 0.2 * keyword`. Копию он клал в `.data/wikilink-graph.json`
— по его собственным словам, «для инспекции»: производный артефакт,
пересоздаваемый из заметок в любой момент. А заметки — ровно то, что Keel
сохранил.

Keel выбросил этот скорер, а не граф. И пользу скорера так и не показали:
весь поисковый стек упирался в потолок recall@5 = 1.0 на золотом наборе из
6 запросов, а на насыщении заслугу нельзя приписать ни графовому
слагаемому, ни любому другому. Рёбра уцелели, скорер — нет, и вклад скорера
остался неизмеренным, потому что eval был насыщен.

По ссылкам теперь ходит модель: правило 2 контракта велит читать индекс
`memory/MEMORY.md` и идти по ссылкам, подходящим к задаче. Ни шага сборки,
ни демона. Утверждать, что это не хуже выброшенного скорера, здесь не
станем: сравнивать было нечем, пока eval стоял на потолке.

## Что Keel сохранил

Четыре вещи заслужили своё место, и только они уцелели:

- Файлы как единственная общая правда: `memory/`, `BACKLOG.md`,
  `PARKED.md`, `OPS.md`, `stages/`. Грепается, диффается, работает без
  демонов.
- Два хука, остановившие реальные инциденты: страж утечки секретов и
  предохранитель от форк-бомбы dev-сервера (Next.js + Turbopack
  форк-бомбили Mac; этот антипаттерн тоже лежит в памяти).
- `verifier` как независимый судья. Самоотчёт — заявление, а не вердикт;
  единственный мультиагентный паттерн, который стабильно окупался.
- Скиллы с ленивой загрузкой: 37 штук — 7 базовых (`stage`, `qa-browser`,
  `audit`, `remember`, `recall`, `safe-dev-server`, `migrate`) и 30
  доменных. Каталог перенесён от предшественника и очищен от его машинерии;
  `recall` — новый. В контекст скиллы попадают только при использовании.

И два добавления. `OPS.md`, доска дежурств: постоянные обязанности с
периодичностью, каждая привязана к скиллу, в режиме `build` или `live`.
Именно она превращает владение продуктом от начала до конца из декларации в
маршрутизируемую работу. Второе — якоря по коду, скилл `recall`, о нём
выше.

## Что НЕ измерялось

Сказано прямо, потому что сравнение, скрывающее свои дыры, — это маркетинг:

- Лобового бенчмарка на задачах не было. Никто не прогонял фиксированный
  набор задач под обоими ядрами со сравнением токенов, времени и доли
  успеха. Все числа в таблице — структурные замеры (размеры, количества,
  строки) либо инцидент в проде с датой; контролируемым экспериментом не
  является ни одно.
- Оценки токенов — именно оценки: из подсчёта символов по ~4 символа на
  токен, без токенизатора.
- «Лучше» здесь значит: меньше грузить, меньше чему доверять, меньше чему
  ломаться. Это не значит «пишет код лучше» — такому утверждению нужен тот
  самый бенчмарк, а его не было.

Что действительно установлено: Keel грузит в 3,6 раза меньше контракта на
каждой задаче, просит доверять в 3,7 раза меньшему объёму кода, обходится
без рантайм-сервисов там, где SkillForge требовал два, и убирает ровно тот
компонент, который OOM-убил живой стейдж, — компонент, которого собственный
рабочий паттерн предшественника уже велел агентам избегать.

## Как воспроизвести числа

```sh
# всегда загруженный контракт
wc -lc skillforge-project/.claude/CLAUDE.md skillforge-project/.claude/_protocol.md
wc -lc keel/bundle/.claude/CLAUDE.md

# код ядра
find skillforge -name '*.ts' -o -name '*.js' -o -name '*.sh' \
  | grep -v node_modules | xargs cat | wc -l
find keel -name '*.sh' -not -path '*/dist/*' | xargs cat | wc -l

# занимаемое место
du -sh skillforge && find skillforge -type f | wc -l
du -sh keel/bundle && find keel/bundle -type f | wc -l

# золотой набор, ради которого строился весь поисковый стек
python3 -c "import json; print(len(json.load(open('skillforge/retrieval-eval.golden.json'))))"

# граф заметка↔заметка: 222 заметки, 919 рёбер, 0 битых ссылок, 4 сироты
bash .claude/skills/memory-consolidation/graph.sh

# память называет код, но код не мог дотянуться до памяти: 141 заметка из 222,
# 494 упоминания, 237 уникальных файлов — одна команда, один набор расширений
EXT='\.(ts|tsx|js|jsx|sql|py|go)'
grep -rlE "$EXT" memory --include='*.md' | grep -v MEMORY.md | wc -l   # заметок
grep -rhoE "[A-Za-z0-9_/.-]+$EXT" memory --include='*.md' | wc -l      # упоминаний
grep -rhoE "[A-Za-z0-9_/.-]+$EXT" memory --include='*.md' | sort -u | wc -l
```
