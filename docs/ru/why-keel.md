# Почему Keel, и что изменилось относительно SkillForge

Keel — преемник SkillForge: тот же автор, та же задача, противоположный вывод.
SkillForge пытался дать агенту *систему*: свой MCP-сервер, векторный индекс по
памяти проекта, реранкер, агентов-персон, гейты согласования, артефакты на
каждую задачу, апдейтер. Keel оставляет четыре вещи, которые действительно
несли нагрузку, и выбрасывает остальное.

Здесь сказано, что именно измерено, что наблюдалось в проде и — что важнее —
**что измерено НЕ было**. Каждое число ниже воспроизводится из двух деревьев;
команды в конце.

## Замеры

Сравниваются SkillForge 1.8.2 в том виде, в каком он был развёрнут в проекте
ShipPulse, и Keel 1.5.0. Токены — оценка из расчёта ~4 символа на токен, это
явно помечено; всё остальное — точный подсчёт.

| | SkillForge 1.8.2 | Keel 1.5.0 | Разница |
|---|---|---|---|
| Всегда загруженный контракт | `CLAUDE.md` (80 строк) + `_protocol.md` (247 строк) = **327 строк / 15 245 символов** (~3,8k токенов, оценка) | `CLAUDE.md` = **81 строка / 4 227 символов** (~1,1k токенов, оценка) | **в 3,6 раза меньше** |
| Код ядра, которому надо доверять и который надо поддерживать | **6 316 строк** TS/JS/shell в 17 файлах — MCP-сервер, конвейер эмбеддингов, eval поиска, sync, update, verify | **1 460 строки** shell в 9 файлах — установщик, сборщик архива, 3 хука, safe-run, зачистка миграции, граф памяти, якоря по коду | **в 4,3 раза меньше** |
| Требуемые сервисы в рантайме | **2** — MCP-сервер (bun) + демон Ollama с эмбеддингами `bge-m3` | **0** | — |
| Занимаемое место после установки | **63 МБ / 3 796 файлов** (24 МБ — `node_modules` MCP-сервера; остальное — векторный индекс и `.tgz`-архивы, которые апдейтер делал *сам себе*). Чистое поставляемое дерево без этого: 4,3 МБ / 133 файла | **304 КБ / 54 файла** | **в 14 раз** меньше самой щадящей меры; **в 212 раз** меньше того, во что он превращается в рабочем проекте |
| Сабагенты | **8 персон** — orchestrator, product, qa, security, devops, research, copy, skill-creator | **2 по изоляции контекста** — `scout` (разведка только на чтение), `verifier` (независимый судья) | — |
| Поиск по памяти | эмбеддинги (`bge-m3` через Ollama) + реранкер + PageRank по графу заметок | индекс `MEMORY.md` + `grep`, плюс якоря по коду (`recall`) | — |
| Золотой набор, ради которого всё это строилось | **6 запросов**, насыщение на recall@5 = 1.0 | — | — |
| Церемония до первой строки кода | протокол + `plan.json` + цитаты `because[]` + гейт согласования — одинаково для переписывания архитектуры и для правки отступа | **0 файлов** на малой задаче; **2 файла** (бриф, отчёт) на большой | — |

**Ядро растёт от версии к версии — и таблица об этом говорит.** В 1.3.0 было
893 строки shell в 5 файлах: перевес в 7,1 раза. В 1.4.0 добавились зачистка
миграции, проверка обновлений и граф памяти — 1 305 строк в 8 файлах, перевес
упал до 4,8 раза. В 1.5.0 пришли якоря по коду — 1 460 строк в 9 файлах,
перевес 4,3 раза. Каждая добавленная строка — это строка, которой кому-то
придётся доверять; а сравнение, которое только улучшается, — это сравнение,
которым управляют.

## Что сказал прод

Числа объясняют цену. Ущерб объясняют четыре заметки — все из собственной
памяти ShipPulse, написанные тогда, когда SkillForge был живой системой. Это
первоисточники, а не реконструкция задним числом.

**1. Инфраструктура поиска убила ту самую работу, которой служила.**
`antipatterns/skillforge-mcp-misreports-ollama-when-direct-call-works.md`: на
macOS Apple Silicon `OLLAMA_NUM_PARALLEL=4` × Metal-хипы `bge-m3` при пачечной
эмбеддинг-нагрузке разносили VM commit → ОС убивала Ollama по OOM → отказ
каскадом уходил в IDE-хост. Это **убило стейдж 009 ShipPulse на середине
фан-аута 31.05.2026 и потеряло три артефакта сабагентов.** Система памяти
уничтожила работу, ради запоминания которой существовала.

**2. Собственный рабочий паттерн системы обходил её собственный поиск.**
`patterns/stage-012-013-multiwave-ship-pattern.md` — паттерн самого успешного
релизного прогона SkillForge — предписывает дословно: *«сабагенты НЕ вызывают
MCP-память (путь через ollama нестабилен); они грепают `memory/**`»*, а заметки
памяти *«пишутся напрямую файлами (MCP `memory_write` отдаёт 500 при
деградировавшей ollama)»*. Векторный индекс, реранкер и PageRank обходились
**лучшей практикой самой системы** — в пользу grep и обычных файлов. Keel — это
тот самый обход, возведённый в архитектуру.

**3. Гейты обходили под давлением — и давление создавала сама конструкция.**
`antipatterns/engine-bypass.md` фиксирует прогоны, в которых пропускали гейт
согласования и помечали юниты зелёными без артефактов, потому что *«под
давлением сроков гейты ощущаются бюрократией»*. Гейт, который обходят на важных
прогонах, — не механизм безопасности, а налог на неважные.

**4. Персоны постоянно всасывали работу, которую должны были лишь маршрутизировать.**
`antipatterns/fat-agent-prompt.md`: файлы агентов приходилось держать в узде
жёстким лимитом в 55 строк с проверкой в `verify`, потому что они непрерывно
обрастали доменной логикой и расходились с каталогом скиллов. Роли требуют
надзора; изоляция контекста — нет. Keel декомпозирует по контексту, а не по
должности, — совпадая с замером самой Anthropic: агенты-роли тратят на
координацию больше токенов, чем на работу.

## Знание, привязанное к коду

Новое в 1.5.0 — скилл `recall`, якоря по коду.

До него заземление в памяти работало **по симптому**: грепнуть `memory/` по
словам из ошибки и надеяться. **По месту** оно не работало никогда. В реальной
памяти проекта из 222 заметок 141 называют файлы кода — **494 упоминания, 237
уникальных файлов**, — и ничто из этого не было достижимо со стороны кода.
Спросить «а что проект знает про `apps/web/proxy.ts`?» *до* того, как открыть
этот файл, было нельзя.

Якорь — это объявление в front-matter: заметка сама говорит, о каком коде она.

```yaml
code:
  - apps/web/proxy.ts#handleRequest
  - apps/web/middleware.ts
```

Отсюда два запроса:

```sh
bash .claude/skills/recall/anchors.sh apps/web/proxy.ts   # что мы знаем про этот код
bash .claude/skills/recall/anchors.sh --check             # мёртвые якоря
```

Ответ приходит двумя наборами. **ANCHORED** — заметки, объявившие этот код в
front-matter: точные и проверяемые. **MENTIONED** — заметки, называющие его в
прозе, ранжированные по плотности: полезно, но шумно и непроверяемо — что и есть
аргумент *в пользу* якорей.

**Ради этого всё и сделано: детекция гнили.** Переименуйте символ или удалите
файл — и `--check` доложит `DEAD_SYMBOL` / `DEAD_FILE`: заметка теперь врёт, и
это видно. Упоминание в прозе проверить нельзя в принципе. Проверено
end-to-end: и переименование, и удаление были пойманы.

**Рёбра код→код здесь не строятся, и это осознанно.** Вызывающих, ссылки и
иерархию вызовов считает serena (LSP, уже прописан в `.mcp.json` из seed) —
`find_symbol`, `find_referencing_symbols` — точно и вживую. Структура кода,
которую поддерживают руками, гниёт; LSP — нет. Слой якорей несёт только то,
чего LSP знать не может: что мы **узнали** про это место в коде.

И правило 2 контракта туда маршрутизирует: *«Прежде чем менять файл, который ты
не только что написал, заземлись по месту: скилл `recall`»*. Именно поэтому
контракт вырос с 79 строк до 81. Скилл, в который никто не маршрутизирует, — это
скилл, которым никто не пользуется; ровно так у предшественника и умер каталог
скиллов.

### А что с графом заметок?

Два графа легко перепутать, и в 1.4.0 этот документ их перепутал — и продавал
тот, который значит меньше. Разделим. Граф **заметка↔заметка** — это гигиена
памяти, и только она. Граф **знание↔код** (выше) — то, ради чего всё делалось.

Строка про поиск в таблице выше — про *скорер*, а не про граф. Рёбра на месте:
это `[[wikilinks]]` внутри `memory/*.md`, там же, где они были всегда. В реальной
памяти проекта из 222 заметок (плюс индекс `MEMORY.md`) — **919 рёбер**, **0
битых ссылок**, **4 сироты**: запустите `graph.sh`, и он напечатает ровно эти
числа. Снос MCP-сервера не изменил в этом ничего.

```sh
bash .claude/skills/memory-consolidation/graph.sh           # хабы, битые ссылки, сироты, итоги
bash .claude/skills/memory-consolidation/graph.sh --edges   # сырой список смежности
```

Но честно про то, чем этот инструмент является: это **гигиена памяти**, и только.
Хабы, битые ссылки, сироты. Писать код граф заметка↔заметка не помогает и не
обещает. Граф, который несёт настоящую нагрузку, — это код ↔ знание, то есть
`recall` выше.

**Граф SkillForge был точно таким же — заметка↔заметка; про код он не знал
вообще ничего.** И источником правды он не был: SkillForge *пересобирал* его на
каждом поиске из тех же `[[wikilinks]]` в тех же файлах (`retrieval-eval.ts`,
`buildLinkGraph`) и прогонял по нему персонализированный PageRank —
«HippoRAG-lite» — как множитель-бустер: `1 + 0.2 * graph` поверх
`0.8 * vector + 0.2 * keyword`. Копию он и правда клал на диск, в
`.data/wikilink-graph.json`, — по его собственным словам, «для инспекции»:
производный артефакт, пересоздаваемый из заметок в любой момент. А заметки —
это ровно то, что Keel сохранил.

**Keel выбросил этот скорер, а не граф.** И польза скорера так и не была
показана: весь поисковый стек упирался в потолок — **recall@5 = 1.0 на золотом
наборе из 6 запросов**. На насыщении заслугу нельзя приписать ни графовому
слагаемому, ни любому другому. Рёбра уцелели, скорер — нет, а вклад скорера
остался неизмеримым, потому что eval был насыщен.

**По ссылкам теперь ходит модель.** Правило 2 контракта предписывает читать
индекс `memory/MEMORY.md` и *«идти по ссылкам, которые подходят к задаче»*. Ни
индекса, ни демона, ни шага сборки. Утверждать, что это *не хуже* выброшенного
скорера, мы не станем: сравнить их было нечем — eval упирался в потолок.

## Что Keel сохранил

Четыре вещи заслужили своё место — и только они уцелели:

- **Файлы как единственная общая правда** — `memory/`, `BACKLOG.md`,
  `PARKED.md`, `OPS.md`, `stages/`. Грепается, диффается, без демонов.
- **Два хука, остановившие реальные инциденты** — страж утечки секретов и
  предохранитель от форк-бомбы dev-сервера (Next.js + Turbopack форк-бомбили
  Mac; этот антипаттерн тоже лежит в памяти).
- **`verifier` как независимый судья** — самоотчёт есть заявление, а не вердикт.
  Единственный мультиагентный паттерн, который стабильно окупался.
- **Скиллы с ленивой загрузкой** — 37 штук: 7 базовых (`stage`, `qa-browser`,
  `audit`, `remember`, `recall`, `safe-dev-server`, `migrate`) и 30 доменных.
  Каталог перенесён от предшественника и очищен от его машинерии; `recall` —
  новый. Попадают в контекст только когда используются.

И то, что Keel *добавил* от себя: `OPS.md` — доска дежурств. Постоянные
обязанности с периодичностью, каждая привязана к скиллу, в режиме `build` или
`live`. Именно она превращает end-to-end ownership из декларации в
маршрутизируемую работу. Второе добавление — якоря по коду, `recall`, про них
выше.

## Что НЕ измерялось

Сказано прямо, потому что сравнение, скрывающее свои дыры, — это маркетинг:

- **Лобовой бенчмарк на задачах не проводился.** Мы не прогоняли фиксированный
  набор задач под обоими ядрами со сравнением токенов, времени и доли успеха.
  Все числа в таблице — это *структурные* замеры (размеры, количества, строки)
  либо инцидент в проде с датой. Ни одно из них не является контролируемым
  экспериментом.
- **Оценки токенов — именно оценки**, из подсчёта символов по ~4 символа на
  токен, а не через токенизатор.
- **«Лучше» здесь означает: меньше грузить, меньше чему доверять, меньше чему
  ломаться.** Это не означает «пишет код лучше» — такое утверждение требует
  того самого бенчмарка, а он не проводился.

Что установлено: Keel грузит в 3,6 раза меньше контракта на каждой задаче,
просит доверять в 4,3 раза меньшему объёму кода, не требует ни одного
рантайм-сервиса там, где SkillForge требовал два, и удаляет ровно тот
компонент, который OOM-убил живой стейдж, — компонент, которого собственный
рабочий паттерн предшественника уже велел агентам избегать.

## Как воспроизвести числа

```sh
# всегда загруженный контракт
wc -lc skillforge-project/.claude/CLAUDE.md skillforge-project/.claude/_protocol.md
wc -lc keel/bundle/.claude/CLAUDE.md

# код ядра
find skillforge -name '*.ts' -o -name '*.js' -o -name '*.sh' \
  | grep -v node_modules | xargs cat | wc -l
find keel -name '*.sh' -not -path '*/dist/*' | xargs cat | wc -l

# занимаемое место
du -sh skillforge && find skillforge -type f | wc -l
du -sh keel/bundle && find keel/bundle -type f | wc -l

# золотой набор, ради которого строился весь поисковый стек
python3 -c "import json; print(len(json.load(open('skillforge/retrieval-eval.golden.json'))))"

# граф заметка↔заметка: 222 заметки, 919 рёбер, 0 битых ссылок, 4 сироты
bash .claude/skills/memory-consolidation/graph.sh

# память называет код, но код не мог дотянуться до памяти: 141 заметка из 222,
# 494 упоминания, 237 уникальных файлов — одна команда, один набор расширений
EXT='\.(ts|tsx|js|jsx|sql|py|go)'
grep -rlE "$EXT" memory --include='*.md' | grep -v MEMORY.md | wc -l   # заметок
grep -rhoE "[A-Za-z0-9_/.-]+$EXT" memory --include='*.md' | wc -l      # упоминаний
grep -rhoE "[A-Za-z0-9_/.-]+$EXT" memory --include='*.md' | sort -u | wc -l
```
